# Sprint Planner

Инструмент для планирования спринтов: распределение задач по людям и фазам, визуализация на таймлайне, обнаружение конфликтов по загрузке.

## Команды

```bash
npm run dev       # локальный сервер на http://localhost:5173
npm run build     # tsc + vite build → dist/
npm run lint      # eslint
npm run preview   # предпросмотр сборки
```

## Стек

React 19, TypeScript, Vite, Tailwind CSS v4. Нет бэкенда — всё хранится в `localStorage`.

## Структура

```
src/
  types.ts          — все интерфейсы (Person, Task, Phase, Sprint, PhaseBlock, DayLoad)
  store.ts          — useAppStore() хук; единственное место записи в localStorage
  conflicts.ts      — вся расчётная логика (расписание фаз, конфликты, нагрузка)
  App.tsx           — корневой компонент, роутинг модалок
  components/
    TimelineGrid    — главная сетка таймлайна с drag-and-drop
    LeftPanel       — список задач слева
    TaskEditor      — модалка создания/редактирования задачи и её фаз
    PeopleManager   — модалка управления участниками
    SprintSettings  — модалка настроек спринта
    WorkloadBar     — визуализация загрузки участника
    Modal           — обёртка модального окна
```

## Ключевые решения

**localStorage.** Ключ хранилища — `sprint-planner-v2`. При изменении структуры данных нужна миграция в `loadState()` в `store.ts` (там уже есть примеры).

**Спринты.** Эпоха — `2026-04-13` (первый понедельник). Каждый спринт = 14 календарных дней. Задачи привязаны к спринту через поле `task.sprintStartDate` (ISO-строка). Отображаются только задачи текущего спринта.

**Фазы и расписание.** Расчёт в `computeTaskPhaseSchedule()` (`conflicts.ts`): фазы идут последовательно, выходные пропускаются через `calcEndDay()`. Фаза может иметь `startAfterDays` — явный отступ от старта задачи.

**Review-фазы.** Фаза с label `review` / `ревью` / `code review` — особый случай: она не назначается на конкретного исполнителя, не создаёт конфликтов, отображается на строке предыдущего исполнителя. Определяется функцией `isReviewPhase()`.

**Конфликты.** Конфликт = суммарная нагрузка человека в один день > 1.0 рабочего дня. External-блоки (review, нет исполнителя) в подсчёт не идут. Логика в `computePhaseBlocks()`.

**Нагрузка (`DayLoad`).** `0` = свободен, `1` = занят, `2` = перегружен. Вычисляется в `computePersonLoad()`.

**ID.** Генерируются через `generateId()` из `store.ts` (random base-36).

**Цвета.** Участникам назначаются из `PERSON_COLORS`, задачам — из `TASK_COLORS` (оба массива в своих файлах). Можно переопределить через поле `color` на задаче или фазе.

## Соглашения

- Новые типы → `types.ts`
- Новые расчёты/алгоритмы → `conflicts.ts`
- Мутации стейта только через хуки из `useAppStore()`, напрямую не писать в localStorage
- Новые компоненты → `src/components/`
- Не добавлять бэкенд и серверный стейт — приложение намеренно локальное
